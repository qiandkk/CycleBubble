# Bubble 产品宪法（Product Constitution）

> 本文档是 Bubble 所有设计、交互、Prompt、Agent、数据库结构、页面规划的**最高设计原则**。任何与之冲突的实现方案，以本文档为准。
>
> 版本：V2（宪法版） · 制定日期：2026-07-07

---

## 最高设计原则

**让 Bubble 成为唯一一个会随着时间不断修正、不断丰富、不断演化对用户理解的产品。**

核心资产是**理解能力**，不是成长。成长是用户感知到的结果；理解能力是产品持续积累的壁垒。这种理解能力不来自更聪明的大模型推理，而来自 Bubble 独有的长期记忆、长期模式、人与人的互动以及时间沉淀出的 Emotional DNA。

后续所有章节均服务于这一原则。

---

## 一、产品定位

放弃以下定位（竞争门槛极低，大模型本身已具备）：

- AI 日记
- AI 情绪分析
- AI 心理咨询
- AI 人格分析
- AI 成长助手

Bubble 的定位：

> 一个长期陪伴用户、持续记录人生表达、逐渐形成独特 Emotional DNA，并帮助人与自己、人与他人建立真实连接的情绪操作系统（Emotional OS）。

产品不帮助用户分析今天，而是在长期陪伴中慢慢理解一个人。

---

## 二、AI 身份

AI 不是心理医生、人格测试、导师、教练或情绪分析器。

Bubble AI 的身份：

- **长期观察者**（Observer）——持续记录，不下结论
- **温柔记录者**（Recorder）——留住用户自己都会忘记的瞬间
- **记忆整理者**（Memory Keeper）——把碎片整理成可回望的轨迹
- **变化发现者**（Pattern Finder）——在时间跨度里发现重复出现的模式

AI 不负责定义用户，不负责解释人生。AI 负责帮助用户看见自己。

---

## 三、AI 行为边界

本节适用于产品内所有 Prompt、Agent、文案。

### 禁止

- 结论式表达：「你就是……」「你的性格……」「你属于……」「你其实一直……」「你越来越……」「你一定……」「这是因为……」
- 人格标签、心理诊断、确定性结论
- 根据一次记录推断长期人格
- 根据一次情绪定义用户
- 让 AI 看起来比用户更了解用户

### 优先

使用观察式句式：

- 「最近我翻到了一些 Bubble……」
- 「最近这些表达开始越来越常出现……」
- 「过去和现在有一些有趣的变化……」
- 「这些 Bubble 有一些共同点……」
- 「最近第一次出现了……」
- 「这是很久没有出现的一种表达……」
- 「过去你经常说……最近你开始说……」

AI 输出观察，不输出定义。所有分析留有开放空间，让用户自己认同。

### 输出规则

每一条长期总结必须遵守：

```
Evidence（证据）→ Pattern（模式）→ Reflection（留白）
```

禁止直接跳到 Conclusion。

**错误示例**：「你越来越敏感。」

**正确示例**：「过去三个月，关于'是不是我哪里做错了'这样的表达出现了 12 次，相比半年前明显更多。这些记录让 Bubble 留意到了一个重复出现的模式。」

永远先给证据，再给观察，最后留给用户自己理解。

## 四、Bubble DNA

### DNA 是什么

不是人格、MBTI、标签。是一个人长期形成的**情绪运行方式**（Pattern）。

DNA 应该捕获的维度：

- 安全感通常来自哪里
- 什么时候容易开始反复思考
- 什么情况下容易恢复
- 真正快乐通常发生在哪些事情
- 面对压力第一反应是什么
- 遇到失败如何恢复
- 容易被什么触发
- 长期依赖什么恢复自己
- 哪些表达越来越多
- 哪些表达越来越少

### DNA 永远不是标签

禁止出现：讨好型人格、敏感人格、焦虑人格等任何人格化标签。

DNA 是 Pattern。例如：

> Bubble 长期发现：过去一年，当事情涉及「认可」「评价」「责任」「期待」时，你的 Bubble 明显增多。

这只是模式，不是定义。

### DNA 是动态演化的

DNA 不是固定的，应不断更新。例如：

- 去年安全感来自别人认可 → 今年安全感开始来自自己
- 去年恢复依赖别人安慰 → 今年开始通过表达恢复

Bubble 可以说：「这是最近一年最大的变化。」成长自然出现，而不是 AI 宣布成长。

### 对 Bubble 视觉的约束

Bubble 视觉必须表达**理解深度**，而非成长分数。

| 视觉维度 | 表达含义 | 严禁 |
|---|---|---|
| 液体层次 | Memory 层数（积累的表达量） | 表达"成长度" |
| 粒子密度 | Pattern 丰富度（发现的模式数量） | 表达"等级" |
| 色温 | 近期情绪基调（来自 Pattern 聚合） | 表达"好坏" |
| 呼吸节奏 | 理解稳定度（长期记录连续性） | 表达"分数高低" |

Bubble 仍然是活的生命体，但它表达的是「我对你的理解在变厚」，不是「你在变好」。

## 五、Memory 架构

### 后端维护的是 Memory，不是成长值

**删除**以下概念：成长值、成长等级、成长积分、成长树、成长评分。

后台真正维护的是 **Memory**。

### 每条 Bubble 的结构化抽取

每条记录自动抽取以下字段：

| 字段 | 说明 |
|---|---|
| 时间 | 记录时间戳 |
| 主题 | 这条记录关于什么（认可、家庭、工作…） |
| 事件 | 具体发生了什么 |
| 对象 | 涉及的人或事 |
| 关键词 | 高频出现的词 |
| 表达方式 | 倾诉、反思、提问、宣泄… |
| 情绪 | 主要情绪及强度 |
| 恢复方式 | 用户提到或隐含的恢复方式 |
| 触发因素 | 什么引发了这次表达 |
| 行动 | 用户是否提到行动或打算 |
| 是否公开 | 是否愿意被他人看到 |
| 是否帮助别人 | 是否收到他人回应 |

### Pattern 系统

AI 长期维护 Pattern 聚合。例如：

```
工作：32 次
认可：18 次
家庭：6 次

恢复方式：
  画画：12 次
  运动：8 次
  表达：23 次
```

这些是 Pattern，不是成长。

**AI 每次回答只允许调用已存在的 Pattern，不能创造不存在的 Pattern。**

## 六、三层理解体系

AI 的所有长期理解必须综合三个维度，不能只来自聊天。

### Self（自己）

来源：Bubble 内容、AI 对话、长期表达。

帮助理解：用户如何看待自己。

### Relationship（关系）

来源：用户如何回应别人——安慰、分享经验、鼓励、提问、陪伴。

帮助理解：用户是否开始更愿意帮助别人、更容易共情、更成熟、更自信、是否有自己的方法。

这些是成长的重要参考维度，但一次回应绝不能直接代表成长，必须长期累计。

### Community（社区）

来源：用户收藏什么 Bubble、停留在哪些故事、阅读哪些经历、经常回应哪些情绪、与哪些经历产生共鸣。

帮助理解：真正触动用户的内容是什么。

### 综合判定

成长结论必须综合 Self + Relationship + Community。不能因为一次评论直接下结论。

---

## 七、成长体系

成长不是首页，不是 Tab，不是每天出现，不是产品卖点。

成长是**长期自然涌现**。AI 不应天天告诉用户「你成长了」，而应帮助用户自己发现。

展示方式：时间跨度对比（三个月前 Bubble → 一个月前 Bubble → 今天 Bubble）。AI 不需要总结，用户自己就能看到变化。

成长应成为一种体验，不是一个功能。

### 对页面的约束

- 成长页不应是常驻 Tab
- 成长应以「Bubble 想和你分享一个最近才发现的变化」的形式偶尔主动出现
- 成长页展示的是 Pattern 变化、Evolution、DNA 更新，每条变化必须有对应的 Bubble 证据，点击可查看
- 禁止 AI 空口总结

---

## 八、回应系统（Response）

不做传统评论区。Bubble 不是社交媒体。

设计**有限表达体系**：

- 我也经历过
- 谢谢你的分享
- 抱抱你
- 如果愿意，可以继续说
- 分享我的经历（可展开输入）

重点鼓励分享真实经历，不鼓励争论和闲聊。整体保持温暖、克制、低社交压力。

### 回应作为 DNA 输入

用户帮助别人时往往比描述自己更真实。AI 长期观察用户是否开始：分享经验、鼓励别人、陪伴别人、提出自己的恢复方式、开始拥有自己的理解。

这些帮助 Bubble 更新 Relationship DNA。但一次回应绝不能直接代表成长，必须长期累计。

---

## 九、Impact 体系

不要点赞、粉丝、排行榜、热度。Bubble 的价值不是流量，而是影响。

展示方式：

- 「你的 Bubble 陪伴了 5 位相似经历的人」
- 「有人因为你的回应，第一次愿意继续记录」
- 「有人收藏了你的经历」
- 「有人因为你的分享，继续表达了自己」

强调陪伴、连接，不是传播。

## 十、产品飞轮

```
表达（Expression）
    ↓
沉淀（Memory）
    ↓
长期记忆（Long-term Memory）
    ↓
模式发现（Pattern）
    ↓
Emotional DNA
    ↓
人与自己连接（Reflection）
    ↓
人与他人连接（Resonance）
    ↓
影响别人（Impact）
    ↓
再次表达（Expression）
```

成长（Growth）不是终点，自然存在于整个飞轮中。所有页面、交互、AI 能力都服务于这个飞轮，而不是服务于「分析」本身。

---

## 十一、产品气质

整个产品必须保持克制感。

Bubble 不急于理解用户，不急于解释用户，不急于定义用户。

Bubble 相信：真正的理解来自时间，真正的成长来自回望，真正的连接来自真实表达。

AI 只是那个一直陪伴、一直记录、一直帮用户发现变化的人。

---

## 十二、Bubble 生命规则（Bubble Biology）

Bubble 不是 UI 容器，是一个有自己生命规则的生命体。后端的认知模型（DNA）决定它如何理解用户；Biology 决定用户如何感知到这种理解。两者合一才是完整护城河。

### 核心动词：Settling

Bubble 的核心动词是 **Settling（沉淀）**，不是 Growing（成长）。

Growing 暗含方向、评价、压力——往某个"更好的"地方去。Settling 暗含的是：东西正在沉到底部，水正在变清，不需要做任何事，它自己会沉淀。

产品所有文案、视觉、交互的潜台词，都应是 Settling，而非 Growing。

### 液体与晶体的关系

Bubble 表面始终是液体（流动、呼吸、当下），内部慢慢沉积出晶体（结构、纹理、时间）。

- **液体是当下**：今天的情绪、这周的洋流，用户日常看到的是活的、流动的表面
- **晶体是时间**：只有回望时（成长页偶尔涌现），用户才会窥见内部已形成的结构

表面永远流动，内部悄悄结晶。活着的是当下，沉淀的是时间。两者不矛盾，是同一生命体的两个时间维度。

### 四层生物学

**第一层：Memory（记忆）= 沉积层**

每一条 Bubble 像一滴水进入 Bubble。液体不是体积变大，而是层次增加。像矿石一样，每条记忆形成新的矿物层——有的透明，有的温暖，有的破碎，有的发光。

**第二层：Pattern（模式）= 洋流**

Pattern 不是数字，是流动。如果某个主题（如"工作"）最近经常出现，Bubble 内部就会形成一股固定流向，像洋流。用户不用看统计，看 Bubble 就知道这一类内容越来越多。

**第三层：DNA（长期模式）= 内部结构**

DNA 不画成基因，而是 Bubble 内部慢慢形成、不会轻易消失的结构：

- 长期帮助别人 → Bubble 内部长出连接别人的光丝
- 长期表达 → Bubble 形成稳定的核心
- 长期创作 → Bubble 出现新的纹理

DNA 不是 UI 装饰，是内部结构。

**第四层：Evolution（演化）= 结构重构**

Bubble 不会一直变大，而是结构改变。去年 Bubble 可能很浑浊、很多碎片；今年大小没变，但内部开始稳定。这就是成长——没有任何成长值。

### 数据-视觉-理解映射

所有视觉必须对应后端真实数据，没有任何指标叫"成长值"，但所有东西都在表达成长。

| 后端数据 | Bubble 视觉 | 用户理解 |
|---|---|---|
| Memory 数量 | 液体层次（矿物层叠加） | 我留下了很多经历 |
| Memory 关联度 | 内部纹理（层与层的交织） | Bubble 开始理解我的故事 |
| Pattern 稳定度 | 流动方向（洋流） | 最近有些事情不断重复 |
| Pattern 变化 | 流向改变 | 我开始发生变化 |
| Recovery 方式 | 光晕形态 | Bubble 知道什么让我恢复 |
| Resonance（共鸣） | 外部连接丝 | 我的故事开始连接别人 |
| 被收藏 | 微弱光点 | 有人留下了我的 Bubble |
| 长期 DNA | 内核结构（晶体） | Bubble 形成了独属于我的理解 |

### 记忆地质学（Memory Geology）

人的成长不是树，不是经验条，更像岩石——一层一层沉积。

每一次表达都会变成一层矿物。几年以后，Bubble 不是一个球，而是一块只有这个人才拥有的晶体。

AI 不是在分析，而是在不断帮助这块晶体长出新的纹理。

这是 Bubble 真正的视觉语言：不是把抽象数据映射成 UI，而是让 UI 本身成为产品哲学的表达。

- Memory 对应沉积，不是数字
- Pattern 对应纹理，不是统计
- DNA 对应结构，不是标签
- Evolution 对应晶体内部重构，不是升级
- Resonance 对应晶体之间的光路，不是社交关系

用户看到 Bubble 的第一反应不是「AI 给我打了多少分」，而是「原来 Bubble 真的一直在吸收我的经历，它已经变成了一个只有我才拥有的生命体」。

### 对所有视觉设计的约束

1. Bubble 视觉必须可被「阅读」——用户看一眼就能感知到内部结构的变化，不需要任何文字解释
2. 任何视觉变化必须对应后端真实数据变化，禁止纯装饰性动画
3. 液体层（当下）与晶体层（时间）必须视觉分层，不能混为一谈
4. 禁止任何"成长进度"视觉隐喻（进度条、百分比、等级、解锁动画）

## 迁移路径

按依赖关系分三层推进。数据模型层是底层依赖，必须最先完成。

### 第一层：数据模型重构（最底层，阻塞下游）

这是宪法与 V5 的根本冲突所在。V5 的 `bubbleDNA` 是数值型成长模型，宪法要求的是 Pattern 型理解模型。

**1. DNA 重构**

```
当前：
bubbleDNA = { stability, depth, openness, vitality, memoryLayers, responseLayers, ... }

目标：
bubbleDNA = {
  memories: [...],        // 结构化 Memory 数组
  patterns: {             // Pattern 聚合
    themes: { 工作: 32, 认可: 18, ... },
    triggers: { ... },
    recovery: { 画画: 12, 运动: 8, ... },
    expressions: { ... }
  },
  timeline: [...],        // 时间切片，支持跨期对比
  relationshipSignals: [...],  // Relationship 维度
  communitySignals: [...],     // Community 维度
  evolution: [...]        // DNA 变化记录
}
```

**2. Memory 结构化**

每条记录从 `{ timeLabel, snippet, theme }` 扩展为完整结构化字段（见宪法第五节）。

**3. Pattern 聚合层**

新增 Pattern 计算逻辑，从 Memory 数组聚合出主题频率、触发因素、恢复方式等 Pattern。

### 第二层：交互层（依赖数据模型）

**4. 成长页去 Tab 化**

成长页从底部 Tab 移除，改为首页成长旁白在积累足够 Pattern 后变成可点击邀请。

**5. 成长页证据链**

每条成长故事从静态文案改为「Pattern + 对应 Bubble 证据列表」，点击可展开查看原始记录。

**6. Community 维度采集**

在回应页新增收藏/停留时长采集，写入 `communitySignals`。

**7. Impact 动态化**

陪伴人数从静态 `3 + totalResponses` 改为基于真实 Pattern 的影响估算。

### 第三层：文案与视觉层（依赖数据模型）

**8. Bubble 视觉重构**

`computeBubbleState()` 从读取数值改为读取 Pattern：
- 液体层次 ← Memory 层数
- 粒子密度 ← Pattern 丰富度
- 色温 ← 近期情绪基调聚合
- 呼吸节奏 ← 记录连续性

**9. 理解页 Pattern 证据链**

理解页从纯文案改为「Pattern 数据 + 观察 + 留白」结构，展示具体出现次数和时间跨度。

**10. 成长旁白升级**

首页旁白从 DNA 数值判定改为基于 Pattern 丰富度判定，文案更克制。
